Old Version Learning Journal (Original)

LinuxHelper is one of my first projects. It was written in Python, which is one of the most widely known languages. The graphical interface library is "Tkinter". My purpose in making this project was to facilitate Windows users' transition to Linux and open new doors for them.

LinuxHelper is Born

After going through a challenging but equally enjoyable learning process about Linux, installing the necessary software tools, and learning Python from the internet, I decided to start making real projects. For my project to be attention-grabbing, my area of interest also had to be attention-grabbing. I learned from some platforms (Reddit, YouTube..) that many Windows users today want to switch to Linux but don't have enough technical knowledge and need help, so I decided to develop software that would assist them. The idea for LinuxHelper was born then.

Main Engine (main.py)

I started making LinuxHelper first with the main engine (main.py). This would contain LinuxHelper's pages, manage distro URLs, define the computer's operating system and download folder, download ISOs from my Google Drive account, write them to USB, and at the same time provide logs while doing this and apply basic operations like these. In short, it would create the software's skeleton. Operations would take place here. Without this code, LinuxHelper couldn't work because it was both calling the screens and performing the backend task to make LinuxHelper do its job.

Main.py starts by showing the welcome screen to the user. Afterwards, the warning screen comes. Users who accept the warnings are shown the hardware screen, and at the same time, their computers are scanned and hardware is classified. Then, a screen asking personal preferences is shown to the user. Finally, with all this information, 2 or 3 distributions (distros) are recommended to the user. The user selects a distro, and main.py starts the real work by downloading the distro ISO.

ISOs are downloaded from ISO lists with a complementary function (_download_iso). Inside the list are the URL's IDs, and the function places the ID into Google Drive URL format to reveal the final distro URL. The distro is downloaded with this URL. The downloaded ISO is written to the USB formatted with diskpart using "dd", or if not available, manually with diskpart. All operations are reported to the user as logs.

After hardware is classified as "Low, mid-low, mid-high, high", user preferences are taken into account and all are compared with the data in the matrix. In my matrix, there are 2 or 3 distributions for each scenario.

SelfLinux Recommender v2 A Complete Rewrite

The Reddit incident taught me something crucial: people will not trust a program that downloads ISOs from someone's personal Google Drive. And they shouldn't. An ISO is a complete operating system image, if someone tampered with it, they could own your machine. So I started thinking about what a real, trustworthy Linux recommender should look like. The answer turned out to be: rewrite everything LOL. The old LinuxHelper had hardcoded URLs, a two-question preference system, raw dd USB writing, no verification, and... a design that looked like a homework project. SelfLinux Recommender v2 fixed every one of those problems and it taught a lot of new things to me along the way.

What Changed

The rewrite touched every single file. Instead of one monolithic main.py doing everything, the new version is split into focused modules: theme.py for consistent styling, cpu_probe.py / gpu_probe.py / ram_probe.py / storage_probe.py / battery_probe.py for hardware scanning, iso_resolver.py for dynamic URL resolution, sha_verify.py for checksum verification, recommender_engine.py for the scoring engine, usb_writer.py for proper bootable USB creation, and dependency_screen.py to make sure all required Python packages are installed before anything else runs. The old screens (WelcomeScreen, WarningScreen, HardwareScreen, PreferencesScreen, RecommendationScreen) were all redesigned with a cleaner look, and a new DependencyScreen was added as the very first thing the user sees.

Design Philosophy theme.py

One of the first things I did was pull all visual constants out of every screen file and into a single place. In the old LinuxHelper, colors and font sizes were scattered everywhere, and changing one thing meant hunting through five files. The new theme.py is short and deliberate: it defines a COLORS dictionary (background, surface, border, text, muted text, primary blue, success green, danger red, and their background tints), a FONTS dictionary (title, h1, h2, body, body bold, small, mono, button), and two functions that every screen uses.

The apply_theme(root) function configures the ttk.Style object so all Tkinter widgets look consistent. It uses the clam built-in theme as a base (the most customizable one), then overrides button styles (Primary.TButton for the blue action buttons, Secondary.TButton for the outline style secondary buttons, Danger.TButton for destructive actions), progress bar colors, radio buttons, and card frames. This means adding a new screen later is just a matter of using the right style names, no per file color tweaking.

The make_card(parent) helper creates the border and inner frame pattern used everywhere: a thin gray outer frame (acting as the border) with a white inner frame padded on all sides. Every content section in every screen uses this pattern, so cards look identical across the entire app. The design is minimal on purpose: no animations, no glow effects, no shadows beyond the thin border. Clean and readable Instead of pure AI slop.

What I learned: centralizing design tokens is not just about convenience. It changes how you think about adding new screens. When the vocabulary (colors, fonts, card helper) already exists, you spend time on the content, not on fiddling with hex values. Also, Tkinter's ttk.Style system is more powerful than it looks. You can define named styles and map them to different states (active, disabled, pressed) with style.map(), which is how the primary button gets a darker shade on hover without any JavaScript like event handling.

Hardware Probing: The Five Great Probes

The old LinuxHelper scanned hardware by reading a few values from psutil and putting them into a simple "low/mid/high" bucket. It worked™, but it was shallow: no CPU benchmark score, no GPU VRAM detection, no storage type (NVMe vs HDD), no battery awareness. The version 2 probes are each their own module, each cross-platform Windows, Linux, macOS, and each returns a structured dictionary with a normalized score.

cpu_probe.py: Scoring the Brain

The CPU probe returns a CpuInfo TypedDict with fields like model, cores, threads, max_ghz, vendor (Intel/AMD/Apple/ARM), generation (a rough hint like "13th gen Intel" or "Ryzen 5000 series"), and a score from 0 to 100. Getting the model name is platform-specific: on Linux it reads /proc/cpuinfo, on Windows it runs PowerShell's Get-CimInstance Win32_Processor (working with powershell is so annoying if you are a linux user), on macOS it calls sysctl -n machdep.cpu.brand_string. The vendor detection uses keyword matching (brands like "Core", "Ryzen", "Xeon") in the model string. Generation detection uses regex patterns. For example, an Intel model string like "i7-13700K" tells us it's 13th generation, while "Ryzen 7 5800X" tells us it's Zen 3.

The scoring function _score_cpu() combines core count, thread count, and max clock speed into a weighted formula. More cores and higher clocks mean a higher score. A 4-core / 8-thread CPU at 4.5 GHz scores around 55; a 24-thread beast at 5.5 GHz scores in the 90s. The category then maps the score to labels: "Very-low" (0-15), "Low-end" (15-30), "Mid-range" (30-55), "High-end" (55-80), "Extreme" (80+).

What I learned: hardware detection on Windows via PowerShell is VERY surprisingly reliable if you use CimInstance instead of the older WMI approach. Also, CPU model strings are a mess Intel alone has "Intel(R) Core(TM) i7-13700K" and "13th Gen Intel Core i7" and "i7-13700KF" and who knows what else. It's like reading assembly code while researching. (but in the boring way) Writing robust regex for generation detection meant accepting that some models will fall through to "Unknown" and... that's fine at this point.

gpu_probe.py: Detecting the Graphics Card (NVIDIA Shut Up)

The GPU probe detects all graphics cards in the system, estimates VRAM, and categorizes each as strong, mid, weak, or nogpu. On Windows, it uses PowerShell's Get-CimInstance Win32_VideoController (these powershell commands frying me...) which returns both the name and AdapterRAM. On Linux, it combines lspci -mm (what a clean command :D) with nvidia-smi (to (hopefully) get VRAM for NVIDIA cards). On macOS, it uses system_profiler SPDisplaysDataType (apple's commands aren't too innocent too).

The categorization uses keyword matching plus VRAM heuristics: if the name contains "RTX 40" or "RX 7" it's strong; if it contains "GTX 10" or "RX 5" it's mid; integrated Intel graphics are (ofc) weak; and no detected discrete GPU means nogpu. VRAM helps disambiguate. A card with 8 GB+ is at least mid, and 12 GB+ is strong.

What I learned: AdapterRAM on Windows is unreliable for some cards (it may report 0 or a tiny number). The Linux world is cleaner: nvidia-smi gives exact VRAM, and lspci is always available. It's like heaven. The keyword-based categorization is a pragmatic compromise: it won't be perfect for every obscure GPU, but it covers the mainstream ones that 95% of users will have.

ram_probe.py: Memory and Its Details

RAM probing starts with psutil.virtual_memory() which gives total, available, and used percent on all platforms. But the extra details: speed in MHz, DDR type (DDR3/DDR4/DDR5), and number of slots require platform-specific calls. On Windows, Get-CimInstance Win32_PhysicalMemory returns Speed and SMBIOSMemoryType (a number like 26 for DDR4, 34 for DDR5). On Linux, dmidecode -t memory provides the same info but requires root. The probe tries the platform-specific path and falls back gracefully if it fails.

What I learned: dmidecode on Linux requires sudo, and there's no clean way around that without setuid wrappers. The probe documents this clearly: the basic numbers (total, available, percent) always work; the extras are best effort. Also, the SMBIOS memory type codes are not well-documented. I had to look them up in the DMTF spec: 20=DDR, 21=DDR2, 24=DDR3, 26=DDR4, 34=DDR5.

storage_probe.py: NVMe, SSD, or HDD

Storage type matters A LOT for Linux. An NVMe drive means fast I/O and good performance with any distro; a spinning HDD means you want something lightweight so your disk doesn't have a panic attack. The probe uses Get-PhysicalDisk on Windows (which returns MediaType as "SSD" or "HDD" and BusType as "NVMe" or "SATA"), /sys/block/ and findmnt on Linux, and diskutil on macOS. The categorization maps NVMe to "high", SATA SSD to "fast", HDD to "low", and eMMC to "low".

What I learned:There's /sys/block/sdX/queue/rotational (1=HDD, 0=SSD), but NVMe drives appear under /dev/nvme* and don't even have that file in the same way. The probe handles both paths: NVMe by device name pattern, SATA by the rotational flag.

battery_probe.py: Laptop or Desktop?

The battery probe is the simplest: psutil.sensors_battery() returns percent, plugged status, and whether a battery exists. If psutil doesn't support it (some Linux setups), the probe falls back to reading /sys/class/power_supply/BAT*/capacity. The result includes is_laptop, which is just a heuristic: if a battery is present, it's probably a laptop. The recommender engine uses this to boost battery friendly distros when the user cares about battery life.

What I learned: psutil.sensors_battery() is not available on all platforms it's Linux-only in some builds for some reason. The sysfs fallback is essential. Also, "laptop detection" via battery presence is imperfect (some desktops have UPS batteries showing up. I mean I expect everything in everywhere at this point), but it's good enough for recommendation purposes.

Dynamic ISO Resolution: iso_resolver.py

This is the module that directly addresses the (holy) Reddit incident. Instead of hosting ISOs on my GOOGLE DRIVE.... (which nobody trusted... naturally), the resolver goes to the official download servers of each distribution, follows redirects, scrapes directory listings, and returns the real, current download URL. It also returns a checksum URL so we can verify the download afterwards. It fails sometimes but... at least it works™

The architecture is per distro resolver functions. Each distribution has its own download page structure, so a one size fits all approach unfortunately won't work. Ubuntu publishes releases at releases.ubuntu.com with version directories; Fedora uses fedoraproject.org with redirect links; Debian has a mirror list; openSUSE Tumbleweed is a rolling release whose URL changes with EVERY SNAPSHOT...; Pop!_OS uses a predictable pattern based on the codename... I remember myself crashing out "WHY DO THEY MAKE THINGS HARDER" while making the resolver... The RESOLVERS dictionary maps display names to their functions, and resolve_iso(distro_name) dispatches to the right one.

Three helper functions power all resolvers. _http_get(url) does a GET request following redirects and returns the HTML content plus the final URL. _resolve_redirect(url) does a HEAD request first (lighter) and falls back to GET if the server doesn't support HEAD. _scrape_latest(index_url, pattern) fetches a directory listing page, finds all filenames matching a regex pattern, sorts them (preferring amd64 builds), and returns the full URL of the latest match.

For example, the Ubuntu resolver fetches releases.ubuntu.com, finds all version directories (like "24.04.3/"), sorts them newest-first, enters the latest LTS directory, and looks for a file matching ubuntu-VERSION-desktop-amd64.iso. The Fedora resolver follows a redirect from fedoraproject.org/workstation to the actual mirror, then scrapes the resulting page. The Debian resolver fetches the official SHA512SUMS file and extracts the ISO URL from it. Each resolver returns a dict with url, sha_url (checksum file URL), sha_algo (sha256 or sha512), and filename.

What I learned: web scraping is fragile. Mirror page layouts change, and regex patterns that work today might (and do) break tomorrow. The per distro approach is more code but more robust when Ubuntu redesigns their download page, only the Ubuntu resolver breaks. I also learned that many download servers use HTTP redirects (302, 301) to point you to the nearest mirror, and urllib follows these automatically. The key insight was that you don't need to mirror the ISO yourself. The distribution's own infrastructure is the source of truth, and following their redirects is both faster (geographically closer mirror) and more trustworthy (you're getting it from their official CDN).

SHA Verification: sha_verify.py

Downloading an ISO from an official URL is a good start, but how does our very careful user know the file wasn't corrupted in transit or tampered with at the mirror? The answer is cryptographic hash verification. The sha_verify module has two parts: the computation engine and the verification UI.

The computation function compute_hash(file_path, algo) reads the file in 1 MB chunks, feeds each chunk to hashlib, and reports progress via a callback. The chunked approach is important because a Linux ISO is typically 2-5 GB reading it all into memory at once would crash the app. The supported algorithms are MD5, SHA-1, SHA-256, and SHA-512, though SHA-256 is the default.

The fetch_expected_hash(sha_url, filename, algo) function downloads the checksum file from the distribution's server and parses it to find the expected hash for our specific ISO filename. Checksum files come in different formats: some have lines like HASH filename.iso, some use HASH *filename.iso, some have just the hash on a single line. The parser handles all three.

The ShaVerifyDialog is a modal Tkinter window that shows the file path, the algorithm being used, a progress bar, and then the result: either a green "Hashes match" confirmation, a red "Hashes DO NOT match" warning, or an amber "No reference available" notice. It runs the hashing in a thread so the UI stays responsive, and it stores a result attribute (True/False/None) that the installer window reads after the dialog closes.

What I learned: hashlib is beautifully simple. You create a hash object, call update() with chunks, and get the hex digest at the end. The tricky part was the checksum file parsing, because every distribution formats their SHA256SUMS file slightly differently. Ubuntu uses HASH filename, Fedora uses HASH *filename, and some distros just have a bare hash with no filename. The regex ^([0-9a-fA-F]{32,128})\s+\*?(\S+) (it's straight up alien language) covers most cases. I also learned that running a hash computation in the main thread would freeze the entire GUI for 30+ seconds. Threading isn't really optional here.

Recommendation Engine: recommender_engine.py

The old LinuxHelper used a simple lookup matrix: if hardware is "mid" and usage is "office", recommend these two distros. He was a simple guy. But the new engine is score based: every distribution has a feature vector (a dictionary of scored attributes from 0-10), and the engine computes a weighted match score between the user's hardware + preferences and each distro's attributes, then returns the top N matches sorted by score.

The DISTROS dictionary defines 17 distributions, each with a feature vector. The attributes include hardware weights (how well the distro runs on low/mid/high hardware), user facing scores (beginner friendliness, stability, cutting-edge, office, development, gaming, creative, visual polish, privacy, free software purity, LTS score, rolling score, battery friendliness), and metadata (desktop environment, package manager, minimum/ideal RAM, whether it's good for old hardware, and a one line summary).

The recommend(state, prefs, top_n) function takes the hardware state (CPU score, GPU category, RAM, storage category) and the user's preference answers, then iterates over every distro and computes a match score. The scoring works like this: hardware match contributes the largest chunk (the distro's weight for the user's hardware tier, multiplied by 6), then usage (office/development/gaming/creative score times 3), then experience level (beginner friendliness times 2.5 for beginners, cutting-edge times 1.5 for advanced users), then update preference (stability + LTS for stable-preferring users, cutting-edge + rolling for bleeding-edge users), and so on for privacy, free software, battery, Windows like interface, and desktop preference. Penalties are applied for things like being below minimum RAM or the user wanting free software but the distro having proprietary blobs.

The result includes a hardware score (0-100, computed as a weighted average of CPU 30%, GPU 20%, RAM 30%, storage 20%), a tier label ("ultra", "high", "mid_high", "mid", "mid_low", "low", "very_low"), and the ranked list of distro matches each with their score, summary, and a list of human-readable reasons (like "Excellent fit for your hardware" or "Beginner-friendly" or "Rolling release with latest software").

What I learned: designing a scoring system is more art than science. The weights (6x for hardware, 3x for usage, 2.5x for experience, etc.) were tuned by running the engine against known hardware profiles and checking that the results made sense. A 2-core Celeron should get antiX or Lubuntu at the top, not Fedora Workstation. A Ryzen 9 with an RTX 4080 should get Nobara and Pop!_OS, not ReactOS. The penalty system is important too: if the user has 1 GB of RAM and the distro needs 4 GB, that's a -30 penalty, which reliably pushes it below lighter alternatives. The "reasons" list was a late addition but turned out to be one of the most valuable features users want to know why a distro was recommended, not just that it scored high. The penalty stuff reminds me about AI training.

Bootable USB Writer: usb_writer.py

You may ask why didn't I just chose rufus and move on. I don't know the answer too XD

This is probably the most complex module in the entire project, and the one that taught me the most. The old LinuxHelper wrote ISOs to USB using dd. The Unix "disk destroyer" that copies bytes blindly from a file to a device. It works in the sense that the bytes end up on the USB, but it creates a raw image that only boots in one specific way. The new usb_writer.py removes the dd dependency entirely and instead builds a proper bootable live USB: it partitions the device, creates a filesystem, mounts the ISO, copies the files, and installs a bootloader. This is what tools like Rufus and balenaEtcher do under the hood but they are still the GOAT.

Rufus I am terribly sorry btw

Partition Schemes: UEFI (GPT) vs MBR (BIOS)

Modern PCs boot using UEFI firmware, which expects a GPT partition table and a FAT32 partition containing an EFI System Partition (ESP) with a boot loader at \EFI\BOOT\BOOTX64.EFI. Older PCs and some firmware settings use legacy BIOS boot, which expects an MBR partition table with an active/bootable partition and boot code in the Master Boot Record. The USB writer supports three options: "UEFI" creates a GPT table with a single FAT32 ESP partition; "MBR" creates an msdos partition table with a single bootable FAT32 partition; and "Auto" inspects the ISO to see whether it ships EFI boot files and picks accordingly. Practically every mainstream distro from the last decade ships an /EFI directory, so Auto almost always picks UEFI.

I did the auto option "recommended" but if you don't really know what to chose, I would recommend doing a quick research.

ISO Inspection

The inspect_iso_boot_support() function looks inside the ISO without fully extracting it. If 7z or 7za is available, it lists the archive contents and checks for paths containing /EFI/ or bootx64.efi (UEFI) and isolinux or syslinux (BIOS). If 7z is not available and we're on Linux, it mounts the ISO read-only and checks for the /EFI directory and /isolinux directory on the mounted filesystem. If neither method works, it assumes both are supported (the safest default).

Linux USB Writing

On Linux, the writer uses parted for partitioning (creating either a GPT or msdos table, then a FAT32 partition from 1 MiB to 100%), mkfs.vfat for formatting, mount -o loop,ro to mount the ISO read-only, and then copies the files. If rsync is available (and we're not root, which is common when running with via sudo), it uses rsync for the bulk copy with progress reporting. If we're root, it uses a custom _copy_tree_with_progress() function that walks the directory tree, computes the total size first, then copies file by file with 4 MB chunk reads so the progress bar updates smoothly even for large squashfs files. Finally, it installs GRUB: for UEFI, grub-install --target=x86_64-efi --removable; for BIOS, grub-install --target=i386-pc. But here's the key insight: most modern ISOs already ship a working /EFI/BOOT/BOOTX64.EFI file, so if the ISO has an EFI tree, UEFI boot it just works™ after the file copy. No GRUB installation needed. GRUB is a fallback for BIOS or for ISOs without an EFI tree.

Windows USB Writing... and the Bugs I Hit

Windows is where things got really interesting (painful). The writer uses diskpart for partitioning (a command line tool that's been part of Windows since the good old DOS days) (I wasn't even existing in those times but I still say "good old times" for some reason...), PowerShell Mount-DiskImage to mount the ISO as a virtual drive, robocopy to copy the files, and optionally bootsect to write legacy boot code. Sounds simple enough. It wasn't.

Here we go..

Bug #1: diskpart Hanging

The first time I ran the writer on my actual Windows machine, diskpart just... stopped. The log showed $ diskpart /s C:\Users\...\selflinux_diskpart.txt and then nothing. The process was alive but producing no output. After a lot of existencal crisis, I realized that diskpart is an interactive tool. It buffers its output and only flushes when it thinks a human is reading it interactively. My code was reading stdout line-by-line with for line in proc.stdout, which blocks until a full line arrives. But diskpart was buffering all its output until the process ended, so the reader blocked forever waiting for a line that wouldn't come until the process exited, and the process wouldn't exit because... well, it would eventually, but the reader was stuck. Deadlock.

The fix was to add a stream parameter to the _run() helper function. When stream=True (default), it reads line-by-line as before good for long-running commands like rsync or robocopy where you want real-time progress. When stream=False, it uses subprocess.run() which waits for the process to complete, then dumps the entire output at once. Diskpart now runs with stream=False and a pretty generous 180-second timeout. No more deadlock.

What I learned: not all command line tools behave like good Unix citizens. Some are interactive programs that happen to accept a script file, and they buffer output differently than a pipe reading program expects. When a tool hangs, the first thing to check is whether it's a buffering issue, not a logic error.

Bug #2: "convert mbr" Error on Freshly Cleaned Disk

After fixing the hanging issue, diskpart ran to completion but reported an error: The disk you specified is not MBR formatted with exit code 2147942487 (which is 0x80070057, E_INVALIDARG). The error happened on the convert mbr step. Here's the thing: after clean wipes a disk, the disk becomes completely blank no partition table at all. A blank disk defaults to MBR-style behavior. Running convert mbr on it is like telling a blank sheet of paper "please be white" it's already white, and the command fails because it expects to find an existing non-MBR partition table to convert.

At least the fix was simple: remove convert mbr from the script entirely. Only convert gpt is needed, and only when the user chose UEFI scheme. After clean, the disk is already MBR compatible, and creating a partition on it implicitly initializes it as MBR.

What I learned: sometimes the fix is to do less. I was trying to be explicit ("make it MBR!"), but the tool already knew that. Also, diskpart's exit code is unreliable. A step like online disk noerr on an already online disk can set a non zero exit code even though nothing is wrong. I learned not to trust the exit code and instead check for specific error markers in the output text and verify that a drive letter actually appeared afterward.

Bug #3: Python 3.14 Lambda Scope

This was a subtle one. In the _write_thread function in main.py, the exception handler had code like this:

except Exception as e:
  self.after(0, lambda: messagebox.showerror("Write failed", str(e)))

This works fine in Python 3.11 and 3.12, but in Python 3.14 (which I was testing with), it throws NameError: cannot access free variable 'e'. The reason is that Python 3.14 changed the scoping rules for exception variables: the e in except...as e is deleted at the end of the except block, so the lambda which is evaluated later, when self.after fires tries to access a variable that no longer exists.

The fix is to capture the string before the lambda: msg = str(e) then lambda m=msg: messagebox.showerror("Write failed", m). The default argument m=msg is evaluated at lambda definition time, not at call time, so it captures the value before e goes out of scope.

What I learned: Python's scoping rules are not set in stone they evolve between versions. The except...as e deletion behavior was always documented (PEP 3110), but CPython previously kept the reference alive long enough for lambdas to work. Python 3.14 enforced the deletion more strictly. Using default argument capture is a robust pattern that works across all versions.

Bug #4: USB Not Booting: "No Operating System Found"

The most frustrating bug.... After writing, all the files were present on the USB: EFI/BOOT/BOOTX64.EFI, boot/grub2, LiveOS, System. The directory tree looked perfect. But when I rebooted and selected the USB, the firmware slapped on my face "no operating system found." The issue turned out to be a mismatch between the partition scheme and the firmware boot mode. The USB had been written with MBR scheme, but my computer was booting in UEFI mode. A UEFI firmware looking for an EFI System Partition on an MBR disk may or may not find it, depending on the firmware implementation. Some do, some don't. Mine didn't.

The fixes were several. First, Auto mode now prefers UEFI when the ISO ships EFI files since practically every modern distro does, and practically every PC since 2012 boots UEFI by default, this is the right default. Second, I added _find_efi_loader() which verifies that \EFI\BOOT\BOOTX64.EFI actually exists on the USB after the copy, and warns if it doesn't. Third, the robocopy flags were improved with /COPY:DAT /R:1 /W:1 to ensure file attributes are preserved and the copy doesn't hang on locked files. Fourth, the final success message now explicitly tells the user to check their boot menu and disable Secure Boot if necessary, because some distros (Fedora, Ubuntu) ship signed boot loaders but others don't.

What I learned: a bootable USB is not just about copying files. The firmware has expectations about the partition table format, the partition type, and the presence of specific boot files at specific paths. If any of these expectations aren't met, the firmware won't find the operating system, regardless of whether the files are all there. Also: UEFI vs BIOS is not just a "preference". It has to match what your firmware is actually doing. Selecting the wrong scheme will give you a USB that looks perfect but doesn't boot.

Dependency Screen: dependency_screen.py

SelfLinux needs a few Python packages that don't come with the standard library: psutil (for hardware probing), Pillow (for distro logo images), and requests (for more robust HTTP downloads, optional). On a fresh Python installation, none of these are present. The old LinuxHelper just crashed if they were missing. The new approach is to show a dependency installation screen as the very first thing the user sees, before any other screen loads.

The DependencyScreen checks each required package by trying to importlib.import_module() it. If it's already installed, a green checkmark appears. If it's missing, a bullet dot appears and the "Install & continue" button becomes active. Clicking it runs pip install --user PACKAGE in a background thread, streaming the pip output into a log console so the user can see what's happening. If the --user flag fails (some environments don't support it), it retries without it. After all required packages are installed, a "Continue" button appears and the screen auto-advances to the WelcomeScreen after 1.2 seconds.

A critical design detail: dependency_screen.py itself imports nothing heavy at module load time. It only imports tkinter, threading, subprocess, importlib, and theme. All of which are available before the dependencies are installed. The hardware probes (cpu_probe, gpu_probe, and so on) are imported lazily by main.py via _load_probes(), which is only called after the dependency screen finishes.

In main.py, the probe imports start as None:

probe_ram = probe_gpu = probe_storage_type = probe_cpu = probe_battery = None

After the dependency screen calls controller.on_dependencies_ready(), the _load_probes() function does the actual imports. If they fail (e.g., psutil still isn't available for some reason), it returns False and the probes stay as None, which the hardware screen handles gracefully by showing "Scanning..." indefinitely rather than crashing.

What I learned: the "lazy import" pattern is essential for applications that bootstrap their own dependencies. You can't import psutil at the top of main.py if psutil might not exist yet. Setting the module references to None and filling them in later is a simple, readable way to handle this. Also, pip install --user doesn't work in all environments. Some Python installations (like some system Python setups on Linux) don't have a user site-packages directory. The retry without---user pattern is a pragmatic workaround.

Screen Redesigns

WelcomeScreen: The Front Door

The new welcome screen is simpler but more informative. It shows the app name ("SelfLinux Recommender"), a subtitle ("Find the right Linux distribution for your machine"), and three feature bullet points inside a card: Hardware analysis, Smart recommendations, and Verified downloads. Each bullet has a bold title and a one-line description. The "Get Started" button at the bottom leads to the WarningScreen. The old screen was just a title and a button the new one tells the user what the app actually does before they commit.

WarningScreen: Informed Consent

The warning screen now has two cards: a white "What this tool does" card that explains the app's purpose in plain language, and a red-bordered "Data loss warning" card that explicitly states writing to USB will erase all data and that the developer is not responsible. The red border uses COLORS["danger"] as the outer frame background with a pink inner background (COLORS["danger_bg"]) the same pattern as the regular cards but with danger colors. The user can continue or cancel. I learned that warning screens should be honest and specific not "be careful!" but "this will erase ALL data on the USB drive you select."

HardwareScreen: The Dashboard

The hardware screen now shows four cards in a 2x2 grid (CPU, Memory, Graphics, Storage) plus a score banner at the top. The score banner shows the overall hardware score (0-100), the tier description (like "Above average handles modern desktops and most games"), and a progress bar. Each card has a title, body text (with real values like "AMD Ryzen 7 5800X / 8 cores / 16 threads / 4.6 GHz max"), and a category subtitle (like "Category: High-end · Score: 72/100 · Zen 3"). The battery status is appended to the RAM card subtitle. The cards update every 800ms via _tick(), reading from the controller's state dictionary, which is populated by the async probe threads. A "Re-scan" button lets users refresh if a device was plugged in later.

What I learned: Tkinter's grid geometry manager is better than pack for 2x2 layouts. Pack is great for vertical stacking, but for side by side cards you need grid with columnconfigure and rowconfigure. Also, the 800ms polling loop is simpler than trying to use event-based updates across threads since Tkinter isn't thread-safe, the probes write to a shared dictionary and the UI reads from it on the main thread.

PreferencesScreen: Ten Questions Instead of Two lol

The old LinuxHelper asked two questions: "Is visual design important?" and "What's your primary use case?" The new screen asks ten, each in its own card, scrollable vertically: usage (office/development/gaming/creative), experience level (beginner/intermediate/advanced), visual polish preference, Windows-like interface preference, update preference (stable/balanced/cutting-edge), gaming intensity (none/casual/serious), battery priority, privacy importance, free software only, and preferred desktop environment (a dropdown with any/GNOME/KDE/Cinnamon/XFCE/LXQt/Pantheon). The questions are defined as a data structure (QUESTIONS list of dicts) rather than hardcoded UI code, which makes it easy to add or reorder questions later.

The scroll implementation uses a Canvas with a scrollbar and an inner Frame. A well known Tkinter pattern for scrollable containers. The inner frame is placed as a canvas window item, and the canvas scroll region is updated whenever the inner frame's size changes. Mouse wheel scrolling is bound globally, which works on Windows.

What I learned: Tkinter doesn't have a native scrollable frame widget. The Canvas plus inner Frame pattern is the standard workaround, and it works well once you get the configuration right. The tricky part was making the inner frame stretch to the canvas width (so cards don't float in the center of a wide canvas) that required binding to the Canvas's <Configure> event and calling itemconfig on the window item to set its width.

RecommendationScreen: The Tier List with Details

The recommendation screen is split into two columns: the left column shows a ranked list of distro matches (top 8), and the right column shows details for the selected distro. Each list item shows the distro name, a score bar (a thin horizontal bar filled proportionally to the match score), and the one line summary. Clicking an item populates the detail panel on the right, which shows the full name, match score, summary, desktop environment, package manager, minimum and ideal RAM, and the "Why this match" reasons list. A "Download & Install" button at the bottom of the detail panel opens the InstallerWindow.

What I learned: building a two column layout in Tkinter with grid is straightforward, but making a clickable list with custom rendering (not a standard Listbox) requires manually creating Frame widgets and binding click events to every widget in the row (the item frame, the top frame, the bar frame, and all their children). Without binding to children, clicking on the text label inside a row wouldn't register as a click on the row itself.

The Engine: main.py

Main.py ties everything together. It creates the main window, manages the shared state dictionary, coordinates the screen navigation, handles async hardware probing, and contains the InstallerWindow (the download/verify/write flow). The old LinuxHelper had everything in one file; the new main.py is more focused because the heavy logic lives in dedicated modules, but it's still the conductor.

Application State

The self.state dictionary holds everything the screens and the engine need: user preferences (visual, usage, experience, update_pref, privacy, free_software_only, battery_priority, windows_like, gaming_intensity, desktop_pref), CPU data (model, category, score, generation, vendor, max_ghz, cores, threads), RAM data (total_gb, available_gb, used_percent, category, speed_mhz, type), GPU data (name, category, vram_mb), storage data (type, category, size_gb, free_gb), and battery data (has_battery, percent, plugged). Each probe thread writes to its own section of this dictionary, and the UI screens read from it on the main thread. No locks are needed because Python's GIL ensures dict writes are atomic for simple assignments.

Screen Navigation

All screens are created at startup and placed in the same grid cell (row=0, column=0) with sticky="nsew". The show_frame(name) method calls tkraise() on the named frame, which brings it to the top of the stacking order. This is simpler than creating and destroying frames on demand, and it preserves state when the user navigates back. The screen order is: DependencyScreen → WelcomeScreen → WarningScreen → HardwareScreen → PreferencesScreen → RecommendationScreen.

Administrator Relaunch (Windows)

Writing to a USB device requires administrator/root privileges. On Windows, the app checks ctypes.windll.shell32.IsUserAnAdmin() at startup, and if not admin, re-launches itself using ShellExecuteW with the "runas" verb, which triggers the UAC elevation prompt. The original process then exits. This means the user sees a UAC dialog before the app window appears, which is slightly abrupt but necessary the USB write will fail without admin rights.

InstallerWindow: Download, Verify, Write

The InstallerWindow is a Toplevel (popup) window that handles the entire download to USB pipeline. It starts by resolving the ISO URL (via iso_resolver.resolve_iso()), then shows the URL and a "Download ISO" button. Downloading uses urllib.request with a 64 KB read buffer, writing to a .part temporary file and renaming it on completion. The progress callback updates the bar and a text label showing downloaded/total/speed/percentage. After download, the user is prompted to verify the checksum (via ShaVerifyDialog). After verification (or skipping it), the user selects a USB drive and a partition scheme (Auto/UEFI/MBR), clicks "Write to USB", and the writer runs in a thread via usb_writer.write_bootable_usb().

The partition scheme selector is a Combobox dropdown populated from usb_writer.available_schemes(), which returns a list of (key, label) tuples like ("auto", "Auto (recommended) detect from ISO"). The UI stores the human readable label and maps it back to the key when the user changes the selection.

USB drive enumeration uses PowerShell on Windows (Get-Disk | Where-Object BusType -eq 'USB') and lsblk on Linux (filtering for removable devices with RM=1). Each detected drive shows its disk number/device path, drive letters, size, and model name.

What I learned: the .part file pattern (writing to a temporary file and renaming on completion) is essential for download reliability. If the download is interrupted, the partial file is easy to detect and clean up. If the download completes and the rename succeeds, you know you have a complete file. Also, urllib.request follows redirects automatically, which is necessary because many distro download URLs redirect to a mirror.

What the _run() Helper Taught Me

The _run() function in usb_writer.py is worth its own section. It's a wrapper around subprocess that handles several concerns: logging the command to the log callback, optionally prepending sudo on Linux, streaming output line by line vs running to completion, and checking exit codes. The stream parameter was the biggest lesson.

When stream=True, it uses subprocess.Popen with stdout=PIPE and reads lines in a loop. This is perfect for longrunning commands where you want realtime output: rsync with --info=progress2, robocopy, grub-install. When stream=False, it uses subprocess.run() with capture_output=True, waits for the process to finish, then dumps all output lines at once. This is essential for diskpart (which buffers output) and also useful for quick commands like wipefs or partprobe where there's nothing useful to stream.

The check parameter controls whether a non-zero exit code raises WriteError. For diskpart, check=False because the exit code is unreliable. For most other commands, check=True. The stdin_text parameter lets you write to the process's stdin (used for some interactive tools). And the timeout parameter prevents hangs diskpart gets 180 seconds, robocopy gets 3600 seconds (large ISOs take a while to copy).

What I learned: wrapping subprocess in a helper function with configurable behavior is one of the best design decisions I made. Without it, every command invocation would have its own error handling, logging, and timeout logic, and the diskpart streaming bug would have been much harder to fix because the behavior would be inconsistent across call sites. With the centralized helper, I fixed the streaming issue once and it applied to every diskpart call.

Lessons From the Whole Project

Building SelfLinux Recommender v2 was a completely different experience from the original LinuxHelper. The first version was about getting something to work at all. The second version was about getting it to work right. Here are the biggest lessons, in no particular order:

Trust the official sources. The Reddit incident wasn't just about Google Drive looking sketchy it was about the fundamental principle that a security tool should download from the distribution's own infrastructure. Dynamic URL resolution with redirect following is more work than a hardcoded URL list, but it means the user is always getting the latest release from the official mirror, and the checksum can be verified against the distribution's own checksum file.

Verify, don't assume. The SHA verification screen adds a step, but it turns a trust question ("is this ISO safe?") into a mathematical fact ("the SHA-256 matches the official checksum" or "it doesn't"). The user can see both hashes side by side and decide for themselves. Without verification, the download is just bytes from the internet.

Score-based systems need tuning, not just design. The recommendation engine's weights (6x for hardware, 3x for usage, 2.5x for experience) aren't theoretical they're the result of running the engine against known hardware profiles and adjusting until the results made sense. A scoring system that gives wrong recommendations is worse than no scoring system at all.

Platform differences are real and painful. diskpart on Windows is not a Unix tool pretending to be one it's a 30-year-old interactive program with its own conventions, and it will deadlock if you treat it like ls. Cross-platform code isn't just about "if Windows do X, else do Y" it's about understanding that the tools on each platform have different assumptions about how they'll be called.

Dependency bootstrapping requires lazy imports. You can't import a module that doesn't exist yet. The DependencyScreen + lazy import pattern solves the chicken-and-egg problem: the screen that installs dependencies can't itself depend on those dependencies.

Bootable USBs are not just file copies. Copying files from an ISO to a USB is necessary but not sufficient for booting. The firmware expects specific partition tables, specific file paths, and specific boot code in specific places. Getting the files there is half the battle; getting the firmware to find them is the other half.

A centralized helper function saves you from yourself. The _run() wrapper in usb_writer.py meant I could fix the diskpart deadlock in one place instead of hunting through every subprocess call. The make_card() helper in theme.py meant every card looks the same without copy-pasting layout code. When you find yourself writing the same pattern three times, pull it into a function.
